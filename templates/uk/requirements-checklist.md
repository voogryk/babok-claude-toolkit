# Чекліст якості вимог (Wiegers)

Використовуйте цей чекліст перед передачею в розробку або підписанням стейкхолдерами.

---

## Якість окремої вимоги

Для кожної вимоги перевірте:

### Зміст
- [ ] **Повна** — Всі необхідні деталі присутні (входи, виходи, поведінка, помилки)
- [ ] **Коректна** — Точно відображає намір стейкхолдера (перевірено з джерелом)
- [ ] **Необхідна** — Прослідковується до бізнес-потреби (є обґрунтування)
- [ ] **Здійсненна** — Технічно досяжна в межах обмежень

### Ясність
- [ ] **Однозначна** — Тільки одна можлива інтерпретація
- [ ] **Без розмитих термінів** — Немає "швидко", "легко", "зручно", "гнучко", "відповідно"
- [ ] **Кількісна** — Вимірювана де можливо (мс, %, кількість)
- [ ] **Узгоджена термінологія** — Використовує терміни глосарію послідовно

### Якість
- [ ] **Перевірювана** — Можна написати тест / є критерії приймання
- [ ] **Пріоритизована** — Має MoSCoW або числовий пріоритет
- [ ] **Простежувана** — Має унікальний ID, джерело та обґрунтування
- [ ] **Атомарна** — Описує одну поведінку (немає "і", що об'єднує кілька вимог)

---

## Якість набору вимог

Для повного набору вимог:

### Покриття
- [ ] Всі бізнес-вимоги мають відповідні функціональні вимоги
- [ ] Обробка помилок специфікована (не тільки happy path)
- [ ] Вимоги безпеки враховані
- [ ] Вимоги продуктивності/масштабованості визначені
- [ ] Вимоги міграції даних існують (якщо застосовно)
- [ ] Вимоги до звітності/аналітики існують (якщо застосовно)

### Узгодженість
- [ ] Немає суперечностей між вимогами
- [ ] Термінологія узгоджена по всьому документу
- [ ] Рівні пріоритетів застосовуються послідовно
- [ ] Нумерація ID послідовна без пропусків

### Повнота
- [ ] Всі потреби стейкхолдерів представлені
- [ ] Всі ролі користувачів покриті
- [ ] Всі інтерфейси специфіковані (UI, API, зовнішні)
- [ ] Нефункціональні вимоги покривають: продуктивність, безпеку, зручність, надійність
- [ ] Обмеження та припущення задокументовані

### Простежуваність
- [ ] Кожна вимога має унікальний ID
- [ ] Існує ланцюжок Бізнес -> Користувач -> Функціональна
- [ ] Матриця трасування актуальна
- [ ] Вимоги-сироти виявлені (без бізнес-обґрунтування)

---

## Типові анти-патерни

| Анти-патерн | Приклад | Виправлення |
|-------------|---------|-------------|
| Позолота | Функція, яку ніхто не просив | Видалити або отримати явне схвалення стейкхолдера |
| Рішення як вимога | "Використати React для фронтенду" | "Фронтенд повинен рендеритись < 2с на мобільному" |
| Складена вимога | "Система повинна X і Y і Z" | Розділити на 3 окремі вимоги |
| Відсутні заборони | Немає вимог "не повинна" | Додати обмеження безпеки/доступу |
| Неявні вимоги | Припускаються, але не записані | Зробити явними, отримати підтвердження |
| Рухома ціль | Вимога змінювалась 3 рази | Зафіксувати скоуп, логувати зміни |
